# Publishing progress to decomp.dev

[decomp.dev](https://decomp.dev) hosts progress dashboards and badges for decompilation projects.
This fork is **prepped** for it; the steps below that **only a human/repo‑admin can do** are listed
explicitly. **Nothing here has been submitted** — no account was created and no project registered.

## What's already in place (done in‑repo)
- **Full‑project coverage** (main segment **+ all 133 overlay modules**). The partial‑link rule is
  patched to emit a per‑module map (`build/partial_<mod>.map`) during `make`, and
  `tools/aggregate_report.py` merges the main map + every module map into one report with a
  **per‑module category breakdown**. Verified locally: 1,239 / 3,954 functions (~31%).
- `scripts/gen_report.sh` — one‑command wrapper that runs the aggregator → `report.json` (decomp.dev
  objdiff‑report format).
- `.github/workflows/progress.yml` — CI that builds the matching ROM and uploads the report as the
  `us_report` artifact on every push to `main`/`master`.
- `report.json` and `build/partial_*.map` are git‑ignored (regenerated by the build / CI).

## ⚠️ Human steps required (in order)

1. **Push this repo to GitHub** (a public repo is required for decomp.dev to read CI artifacts).
2. **Add the base‑ROM secret** so CI can build (the ROM is never committed):
   - Locally: `base64 -w0 baserom.us.z64 > rom.b64`
   - GitHub → repo **Settings → Secrets and variables → Actions → New repository secret**
   - Name: `BASEROM_US_Z64_BASE64`, value: contents of `rom.b64`.
   - *(This stores your ROM as an encrypted CI secret. If you're uncomfortable putting the ROM in CI
     at all, you can instead run `scripts/gen_report.sh` locally and skip CI — but then decomp.dev
     can't auto‑update, so a public dashboard isn't really viable. Your call.)*
3. **Confirm the workflow runs green** — push, check the Actions tab, verify the `us_report` artifact
   is produced and the build SHA‑1 matches.
4. **Register the project**: as the repo admin, go to <https://decomp.dev/manage/new> and add the
   repository. *(This is the submit step — left entirely to you.)*
5. **Install the decomp.dev GitHub app** on the repo (optional but recommended) so it gets notified
   on workflow completion and can comment progress on PRs.
6. **Add a progress badge** to the README (optional) once the project is live:
   ```md
   ![progress](https://decomp.dev/<owner>/<repo>/us.svg?mode=shield&measure=matched_code_percent&label=progress)
   ```

## Coverage (resolved)

The report covers the **whole project** — the main segment **and all 133 overlay modules**. This was
the one real gap: modules are partially linked and the aggregate maps localize their symbols, so they
were invisible to a single‑map report. Fixed by:
1. **Makefile** — the partial‑link rule now emits a per‑module map: `$(LD) -r $^ -Map $(@:.o=.map) -o $@`.
2. **`tools/aggregate_report.py`** — runs `mapfile_parser objdiff_report` on the main map + each
   `build/partial_<mod>.map`, sums the measures, concatenates units, and emits a per‑module category.

Cross‑checked against `tools/progress.py` / `score_functions.py` (which scan the asm tree directly).

## Local usage
```bash
source ./.venv/bin/activate
make -j6                       # builds the ROM AND emits build/partial_*.map
scripts/gen_report.sh          # aggregates main + all modules -> report.json + headline %
```
